- V1 conf: `account_overdraft_history.json`
- 2 data sources: `datalake.overdraft_events`, `datalake.accounts`
- SQL: JOIN overdraft_events to accounts on account_id + ifw_effective_date, SELECT with account_type enrichment
- Output: **Parquet**, 50 parts, Overwrite, 2-3 rows/date (varies)
- Output columns: overdraft_id, account_id, customer_id, account_type, overdraft_amount, fee_amount, fee_waived, ifw_effective_date, etl_effective_date
- Anti-patterns:
  - **AP4**: From accounts: `customer_id`, `account_status`, `interest_rate`, `credit_limit` sourced but only `account_type` is used. From overdraft_events: `event_timestamp` sourced but unused. Remove unused columns from both DataSourcing modules.
- NOTE: This is the CLEANEST multi-source job -- the JOIN is meaningful (enriches with account_type). No dead code, no dead sources, just unused columns. AP4 remediation only.
- Job conf: Use `ParquetFileWriter` with `numParts: 50`. Match V1 naming exactly.
- Proofmark config: `reader: parquet`

---

**PreferenceChangeCount** (abbreviation: PCC):
- V1 conf: `preference_change_count.json`
- 2 data sources: `datalake.customer_preferences`, `datalake.customers`
- SQL: RANK() window (not ROW_NUMBER), GROUP BY customer_id with MAX CASE pivots for email/sms opt-in
- Output: **Parquet**, **1 part** (not 50!), Overwrite, 2230 rows/date
- Output columns: customer_id, preference_count, has_email_opt_in, has_sms_opt_in, ifw_effective_date, etl_effective_date
- Anti-patterns:
  - **AP1**: `customers` table is sourced but NEVER referenced in SQL. Dead-end sourcing. Same pattern as SecuritiesDirectory/TransactionSizeBuckets.
  - **AP4**: From customer_preferences: `updated_date` sourced but unused. `preference_id` only used in RANK ORDER BY. From customers: ALL columns unused.
  - **AP8**: RANK() window function computes `rnk` but it's never used for filtering or output. Dead computation.
- NOTE: Unlike FeeWaiverAnalysis, the `customers` source here is clearly dead -- no JOIN references any customers columns, and there's no join condition that could inflate rows. Safe to remove (AP1 remediation). But like the FeeWaiverAnalysis approach, keeping it is also fine -- the safer option is to keep it and just document.
- CRITICAL: `numParts: 1` (not 50!). This job has only 1 Parquet partition. Match V1 exactly.
- Job conf: Use `ParquetFileWriter` with `numParts: 1`.
- Proofmark config: `reader: parquet`

CRITICAL REMINDERS:
