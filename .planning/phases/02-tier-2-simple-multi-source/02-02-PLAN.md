**CardAuthorizationSummary** (abbreviation: CAUTH):
- V1 conf: `card_authorization_summary.json`
- 2 data sources: `datalake.card_transactions`, `datalake.cards`
- SQL: JOIN on card_id, ROW_NUMBER (unused), `unused_summary` CTE (dead), GROUP BY card_type with approval counts
- Output: CSV, LF, trailer (`TRAILER|{row_count}|{date}`), 2 data rows + trailer
- Anti-patterns:
  - **AP4**: `card_txn_id` from card_transactions and `customer_id` from both sources unused. Remove from DataSourcing columns.
  - **AP7**: Integer division for `approval_rate` produces 0 for all values. THIS IS LOAD-BEARING. Preserve exactly. Document as AP7 but do NOT remediate.
  - **AP8**: ROW_NUMBER in `txn_detail` CTE never filtered on -- remove. `unused_summary` CTE is literally dead code (defined, never referenced) -- remove entirely.
- Proofmark config: `reader: csv`, `header_rows: 1`, `trailer_rows: 1`
- CRITICAL: The trailer format `TRAILER|{row_count}|{date}` is DETERMINISTIC (date-based, no timestamp). Proofmark still needs `trailer_rows: 1` because the trailer is not a data row.
- CRITICAL: Preserve integer division exactly. `CAST(approved AS INTEGER) / CAST(total AS INTEGER)` in SQLite produces 0 when approved < total.

**FeeWaiverAnalysis** (abbreviation: FWA):
- V1 conf: `fee_waiver_analysis.json`
- 2 data sources: `datalake.overdraft_events`, `datalake.accounts`
- SQL: LEFT JOIN overdraft_events to accounts on account_id + ifw_effective_date, GROUP BY fee_waived
- Output: CSV, LF, no trailer, 1-2 rows/date
- Anti-patterns:
  - **AP1**: `accounts` table is sourced with 7 columns but ZERO accounts columns appear in SELECT, WHERE, or GROUP BY. The entire JOIN might be pointless.
  - **AP4**: From overdraft_events: `overdraft_id`, `customer_id`, `event_timestamp` sourced but unused. From accounts: ALL columns unused.
- **CRITICAL INVESTIGATION**: The LEFT JOIN could be load-bearing (same pattern as ComplianceResolutionTime's cartesian join in Phase 1).
  1. First, write the RE job conf WITH the LEFT JOIN intact (matching V1 exactly except for AP4 column removal)
  2. Run it for a single date (e.g., 2024-10-01) and verify output matches V1
  3. Then test: does `account_id` have multiple matches in `accounts` for the same date? Query: `SELECT account_id, ifw_effective_date, COUNT(*) FROM datalake.accounts GROUP BY account_id, ifw_effective_date HAVING COUNT(*) > 1;`
  4. If no duplicates: the LEFT JOIN is truly dead (AP1). Document but keep it for safety (removing changes nothing but adds risk).
  5. If duplicates exist: the LEFT JOIN inflates counts (load-bearing). MUST keep it.
  6. Either way, document the investigation results in the BRD, FSD, and anti-pattern assessment.
- Proofmark config: `reader: csv`, `header_rows: 1`, `trailer_rows: 0`
- The safest approach: Keep the LEFT JOIN in the RE job conf regardless. AP4 remediation (removing unused columns from DataSourcing) is always safe. AP1 (removing the entire accounts DataSourcing) is the risky remediation -- skip it, just document.
